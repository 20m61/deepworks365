# [Security] PoC意思決定フローの本番前提条件(Entra承認者/冪等性/PII保存)

<!-- labels: type:security,priority:p1,area:decision,status:triage -->

## 背景

`feat/decision-poc`（PoC #25, `docs/superpowers/specs/2026-07-11-decision-poc-design.md`）は
Meeting→Decision→Delivery の縦断ドメイン核をオフライン TDD で検証した。台帳側は
構造的ガード（`approved_decision` は `approval` メタ無しに作成不可、`supersedes` は存在確認必須）
で非交渉ルール2/4/7を強制するようになったが、これは in-memory / JSON ファイル台帳の範囲での
保証であり、本番相当の前提条件（実認証、再配信耐性、保存先の保護）は未実装のまま最終レビューで
先送りされた。本番投入前にこれらを解消する必要がある。

## 受入条件

- [ ] `apps/ingest-func` の `POST /api/decisions/{id}/approve` が匿名HTTPを解消し、
      Entra 認証済み呼び出し元の ID に `approver` を束縛する（リクエストボディの
      自己申告 `approver` を信頼しない）
- [ ] Service Bus からの `meeting.ended` 再配信に対して ingest が冪等になる
      （`payload.id` 等での重複排除により、同一 transcript の二重取り込みで
      台帳エントリが重複生成されない）
- [ ] 台帳の保存先を選定し、PII（`approval.approver` を含む）を平文で
      `/tmp` 等に保存しない（暗号化・アクセス制御を備えた永続層へ移行する）
- [ ] 承認済み（`supersedes` されて版が繰り上がった）エントリに対する再承認を
      構造的に拒否するガードを追加する（最新版以外への `approve` を防ぐ）

参照: `docs/superpowers/specs/2026-07-11-decision-poc-design.md`
