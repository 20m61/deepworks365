# [Security] pnpm10サプライチェーン強化を段階導入する

<!-- labels: type:security,priority:p1,area:security,status:triage -->

## 背景

semgrep の `package_managers.*` ルールは、`minimumReleaseAge` / `blockExoticSubdeps` /
`trustPolicy: no-downgrade` (pnpm) および min-release-age (npm) の設定を要求する。
Platform Skeleton (`feat/platform-skeleton`) へ一括ハードブロックとして導入したところ、
release-age隔離とtrust-downgrade判定が依存解決(vitest 3 / vite / esbuild 系のバンプ)と
衝突し、`pnpm install` が失敗するカスケードを起こした。

このIssueでは、`.npmrc` の `min-release-age` と `pnpm-workspace.yaml` の
`minimumReleaseAge` / `minimumReleaseAgeExclude` / `blockExoticSubdeps` / `trustPolicy`
を段階的に再導入するための設計と検証を行う。再導入までの間、
`scripts/run-security.sh` の semgrep 呼び出しから対象4ルールを `--exclude-rule` で
除外している。

## 受入条件

- [ ] 各設定 (`minimumReleaseAge` / `blockExoticSubdeps` / `trustPolicy`) が依存解決・
      CI・ローカル開発に与える影響を評価する
- [ ] dev toolchain (vitest/vite/esbuild等) に対する例外運用方針
      (`minimumReleaseAgeExclude` の対象範囲・レビュー基準) を決定する
- [ ] CIを使わないローカルゲート (`make security` / pre-commit) 上で
      段階導入後の `pnpm install` が失敗しないことを検証する
- [ ] `scripts/run-security.sh` の `--exclude-rule` 除外を解除し、
      semgrep `package_managers.*` の4ルールが再びブロッキングで通ることを確認する
